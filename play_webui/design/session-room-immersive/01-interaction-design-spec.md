# 独立沉浸式 Session 页面交互设计规范

> 状态：设计基线 v1  
> 适用范围：Play WebUI 中独立于标准 SessionRoom 的沉浸式游玩页面  
> 对照原型：本目录下 `index.html`、`styles.css`、`app.js`

## 1. 目的

这套设计新增一个独立的沉浸式 Session 页面，把同一场 Session 投影为“持续运行的角色扮演舞台”。它不替换、不嵌套标准 `SessionRoom` 页面；两者共享会话业务能力，但各自拥有布局、组件树和页面级 UI 状态。视觉语言借鉴 Galgame 的对白与立绘、RPG 的角色与世界状态，但仍以 Play WebUI 的实时自由推演能力为真源。

本规范用于约束后续生产实现，避免出现以下偏差：

- 把自由推演重新包装成固定章节、主线或支线剧本。
- 为了视觉分页而截断、改写或重新保存 assistant 正文。
- 把人物关系简化成单一好感度或固定的 0–100 进度条。
- 为了舞台效果绕过现有 Session、Agent、状态表、媒体或停止生成链路。
- 把模型内部思维过程当作必须持久化、必须展示的产品数据。
- 把沉浸式组件和页面状态继续堆进 `SessionRoom.tsx`，导致标准页面膨胀。
- 为了沉浸式分页新增、修改或重新解释正文 SSE 事件。

## 2. 核心设计原则

### 2.1 页面隔离，共享会话运行时

标准 SessionRoom 与沉浸式页面是两个独立入口，通过共享的前端领域运行时复用业务逻辑：

```text
Standard SessionRoom Page ─┐
                           ├─ Shared Session Experience Runtime
Immersive Session Page ────┘
```

共享运行时负责 Session 数据与 history window、stream/stop、角色绑定、Composer actions、主模型与 Context Preview，以及 status/media/TTS 等领域能力。两个页面分别负责自己的布局、组件树和视觉交互；沉浸式页面独有的对白页状态、焦点角色、HUD、抽屉和纯净舞台状态不得进入 `SessionRoom.tsx`。

这里的“运行时/中间层”是前端领域 provider、hooks 与 service 的组合，不是 Next.js `middleware.ts`，也不负责 HTTP 路由拦截。

### 2.2 舞台优先，操作随叫随到

背景、立绘和当前对白构成默认视觉中心。日志、状态、设置和媒体管理属于工作台能力，通过大抽屉或模态临时覆盖，不长期挤压舞台。

### 2.3 实时推演，不使用章节语义

页面只使用以下稳定概念：

- Session：一次持续推演。
- Turn：一次玩家输入及其响应。
- 当前场景：实时 scene 状态。
- 当前目标、线索、待办：由状态表或派生视图提供。
- 剧情裁定：某个 turn 的 Narrative Outcome。

不得在主 HUD 或日志中恢复“第几章”“主线”“支线”等需要固定剧本的数据语义。若未来存在用户手工书签或阶段摘要，应使用“书签”“阶段回顾”等独立概念。

### 2.4 完整正文是真源，沉浸式对白是投影

带 `<rp-narration>`、`<rp-character name="…">` 标签的 assistant `content` 是正文唯一真源。现有 SSE parser/reducer 先按既有协议组装 canonical assistant content，沉浸式页面再解析段落并按实际视图做视觉二次分页：

```text
现有 SSE 事件
→ 现有 parser/reducer 组装 canonical assistant content
→ parseAssistantTextSegments()
→ 前端视图测量 + 沉浸分页策略
→ 临时视觉页面
```

舞台说话人、旁白、分页和逐字显示都只是前端展示投影：

- 不新增或修改 SSE event type、payload 与完成语义。
- 不按 SSE chunk 边界切页，也不把 chunk 当段落。
- 不截断历史正文。
- 不把 `pageIndex`、`pageCount` 或分页结果写回 SSE、API、消息 metadata、数据库或 localStorage。
- 刷新后可从完整正文和当前视图重新计算页面；视口不同可以产生不同页数，但重组后的正文必须逐字符一致。

### 2.5 状态优先使用通用表，不虚构专用领域对象

当前关系、人物状态、线索、目标和待办均优先由现有状态表表达。UI 可以提供更友好的投影，但没有明确数据标识时必须退回通用状态表展示，不得依靠表名或自然语言猜测业务类型。

### 2.6 故障隔离

背景、立绘、TTS、日志诊断或状态抽屉故障不得阻断核心聊天。角色绑定无效、Context 门禁或主 turn 失败属于核心阻断状态，必须明确展示并提供现有恢复路径。

## 3. 页面层级

独立沉浸式页面从后向前分为六层：

1. **背景层**：当前 Session 背景、过渡和可调暗度遮罩。
2. **角色舞台层**：在场角色立绘、站位、层级、焦点与弱化效果。
3. **环境 HUD 层**：Session 标题、当前场景、声音、状态、日志和舞台菜单。
4. **交互层**：窄幅对白、预设行动、自由行动输入和生成状态。
5. **工作台层**：角色与世界状态抽屉、全部推演日志、设置及媒体相关模态。
6. **强制模态层**：无效玩家角色绑定、删除确认等不可被舞台模式绕开的流程。

层级约束：

- 背景与立绘不接收输入。
- 默认 HUD 不能永久遮住人物面部与上半身。
- 工作台打开时暂停自动推进，关闭后按原状态恢复。
- 强制模态优先于纯净舞台，并保持其既有不可取消规则。

## 4. 默认桌面布局

### 4.1 顶部 HUD

左侧显示 Session 标题和“自由推演”标识，中部显示当前场景，右侧提供声音、角色状态、推演日志和舞台菜单。

顶部不展示 story ID、session ID 等调试信息。需要这些信息时放入设置或诊断详情。

### 4.2 角色舞台

- 在场角色可以为 0、1 或多个，不假定只有玩家与单一攻略角色。
- 当前说话人获得更高不透明度、轻微缩放或光照焦点；其他角色弱化但不消失。
- 多角色布局优先使用可配置站位；没有站位数据时使用确定性、安全降级布局。
- 角色缺少可靠立绘时显示头像卡、剪影或首字母占位，不把普通头像强行拉伸成全身立绘。
- 背景或立绘不可用时保留可读的对话与操作层。

### 4.3 窄幅对白 Dock

对白 Dock 居中贴近底部，但宽度必须明显小于整屏：

- 常规桌面目标宽度约为视口的 58%–62%，并设置像素上限。
- 2K 目标宽度约为视口的 48%–54%。
- 4K 继续设置像素上限，不能随屏幕无限拉长。
- Dock 只遮挡角色下肢或舞台低价值区域，角色头部、表情与大部分上半身必须可见。
- 说话人铭牌与正文分栏；窄屏改为浮动铭牌。

正文允许自然换行。单个展示页面不追求装下完整 assistant 段落，而追求舒适阅读和尽量少遮挡立绘。

## 5. 对白分段与分页

### 5.1 数据流与分段顺序

分页只发生在沉浸式页面的视图层，不改变网络协议或消息真源。生产实现必须按以下顺序生成展示单元：

1. 复用现有 SSE parser/reducer，按现有事件协议组装 canonical assistant `content`；流式期间输入是已组装的 canonical 前缀，完成后是完整正文。
2. 对 canonical content 调用 `parseAssistantTextSegments()`，解析旁白与角色段落。
3. 每个段落保留 `kind`、`speakerName` 和完整 `text`。
4. 根据对白容器实际宽高、字体、字号、行高、段落类型和沉浸分页策略测量并切成展示页。
5. 页面只在内存中保存当前消息 ID、段落索引、页索引和已读边界等临时 UI 状态。

`history-page` 与这里的分页是两件事：前者是服务端按 turn 加载历史窗口，后者是对已加载 assistant 正文做纯前端视觉二次分页。两者不得共用 `page` 参数、状态或语义。

### 5.2 分页规则

- 中文优先在 `。！？；……` 后分页，其次在 `，` 后分页。
- 引号应尽量与其内容保持在同一页；不能保证时允许跨页，但不得丢字符。
- 极长无标点文本按可见字符或测量结果强制换页，避免横向溢出。
- 每页建议 2–4 行；字体放大后页容量自动下降。
- 一条消息可以出现多次“继续阅读”，例如 `1/3 → 2/3 → 点击继续`。
- 点击正在逐字显示的页面，先立即补全当前页；再次点击才进入下一页。
- AUTO 模式在当前页显示完成后计时，不跳过未读页面。
- `prefers-reduced-motion` 下直接显示当前页完整文本，但仍保留分页。
- 容器尺寸、字体缩放、行高、浏览器缩放或屏幕方向变化后重新测量和分页。
- 同一正文在不同视口允许产生不同页数；把所有视觉页按顺序拼回后，必须与输入段落逐字符一致，不能丢失、重复、改写或重排字符。

### 5.3 流式期间的稳定性

- 不等待或要求后端发送“分页完成”事件；现有 `text_delta` 等事件保持原样。
- 不按一次 `text_delta` 或任意网络 chunk 的长度决定页边界。
- 流式正文增长时，已展示和已读页面保持稳定；只对尚未阅读的尾部重新测量与分页，避免当前页跳动。
- 如果视口或字体变化使已读页必须失效，按完整 canonical content 重新分页，并将阅读位置锚定到稳定的字符偏移，而不是旧页码。
- `turn_completed` 后对完整 canonical content 做最终校验；最终页拼接结果必须与完成事件对应的 assistant 正文完全一致。

### 5.4 可访问性

视觉逐字显示元素对读屏隐藏，另设 `aria-live="polite"` 区域一次播报当前完整页面，避免每个字符都触发播报。页面切换后更新播报内容。

## 6. 行动与生成

### 6.1 自由行动

自由行动是默认可靠入口，复用现有 IC、OOC、GM、命令、叙事风格、主模型选择、Context 门禁、发送和停止生成能力。

进入自由行动时：

- 暂时用输入面板替换对白框，而不是另开侧栏。
- 保留当前 mode 和已有快捷配置。
- Enter 发送，Shift+Enter 换行。
- 生成中发送按钮变为停止按钮，并使用现有 requestId 停止链路。

### 6.2 预设行动

原型中的三项分支用于展示 Galgame 式交互，不代表生产系统已经具备“每 turn 动态生成分支”的能力。

生产第一阶段只能映射现有 story quick replies，并明确其是故事配置的快捷输入。真正的动态建议必须等待独立的 turn-scoped 数据契约，不能从 assistant 正文中猜测选项，也不能把选项偷偷写入消息 metadata。

## 7. 纯净舞台

舞台菜单提供“进入纯净舞台”：

- 隐藏顶部 HUD、场景说明、关系入口、选项、对白、抽屉、菜单和 toast。
- 只保留背景与立绘。
- 所有隐藏层同时设置 `aria-hidden` 和 `inert`，不能继续被 Tab 聚焦。
- 始终保留一个明确的恢复按钮，并支持 `H` 和 `Esc` 恢复。
- 进入前关闭工作台与输入面板；退出后焦点回到原舞台菜单入口。
- 纯净舞台是当前页面的临时状态，默认不写 localStorage。
- 强制角色绑定、错误恢复或删除确认不受纯净舞台隐藏。

## 8. 角色与世界状态工作台

状态工作台在桌面使用宽幅抽屉，在移动端使用接近全屏的 bottom sheet。它包含五类信息：

1. 当前场景。
2. 角色卡与各自绑定状态表。
3. 人物关系状态。
4. 普通状态表。
5. 线索和待完成事项。

### 8.1 多角色

- 展示玩家角色、在场 NPC 和相关离场角色。
- 每个角色可以绑定 0–多张 normal 状态表。
- 同一页面可以存在多个关系对象和多个角色状态表。
- 不将“当前聚焦角色”误写成“唯一关系对象”。
- 场景中的在场角色名称仅用于展示；需要稳定关联时使用 character mount ID 契约。

### 8.2 人物关系不是一个数字

关系卡必须支持任意数量与类型的字段，包括但不限于：

- 数字：信任、亲密、依赖、戒备、熟悉、疏离。
- 枚举或短文本：默契期、旧友、敌对、若即若离。
- 长文本：双方当前相处状态、最近变化、未解矛盾、关系备注。
- 非量化值：未知、不可见、尚未确认。

约束：

- UI 不假定所有值都在 0–100。
- UI 不根据某个数字自动推导阶段或关系文案。
- 没有明确单位时不自动画百分比进度条。
- 字段顺序遵循状态表行顺序或明确的投影配置。
- 缺失字段不补默认值，不显示伪造的 `0`。
- 同一次更新可以同时改变多个数值字段和一个或多个文本字段。
- 日志应逐字段显示变化，例如“信任 64→68；亲密 58→60；阶段 默契期→信赖加深；关系描述已更新”。

推荐的现有状态表表达方式是一名关系对象对应一张 normal 状态表，行可以是“信任”“亲密”“阶段”“关系描述”“最近变化”。若未来要生成关系网络专用视图，必须有显式且类型化的关系表标识；没有标识时按普通状态表渲染。

### 8.3 线索与待办

线索和待办当前也视为状态表的产品化投影。没有专用契约时，工作台可按通用表展示，不应通过表名中是否包含“线索”“任务”来决定业务逻辑。

## 9. 全部推演日志

日志只有一个“全部”范围，不提供章节或多 tab。按 turn 展示当前能够获得的记录：

- 玩家输入或预设决策。
- assistant 旁白与角色对白。
- tool call / tool result。
- Narrative Outcome。
- system / command / error。
- 运行期公开的 thinking 摘要或诊断。

信息层级：

- 第一层：来源、turn、时间、简短正文。
- 第二层：工具参数预览、结果预览、usage、错误码等可展开详情。
- thinking 统一称“思考摘要”或“诊断”，不得承诺展示内部完整推理链。
- 刷新后只展示后端实际持久化的内容；没有持久化的运行期 thinking 不伪造、不回填。
- 用户点击预设行动目前仍会形成普通 user message；在没有来源字段前，历史中不得声称能够可靠区分“预设决策”和“自由输入”。

日志抽屉复用现有 `history-page` 的服务端 turn 分页能力，支持加载更早、加载更新和跳到最新。大历史不能一次性载入；该分页只控制加载哪些 turn，与对白 Dock 的前端视觉分页完全独立。

## 10. 字号与响应式

### 10.1 字号

- 沉浸式舞台默认字号不能低于当前 Session 125% 的可读基线。
- 继续复用 90%–200% 用户字体缩放。
- 字体缩放影响对白、日志正文、状态值和输入框，不只影响聊天气泡。
- 小型英文 kicker 只作为辅助装饰，关键信息必须另有可读中文文本。

### 10.2 断点行为

| 环境 | 行为 |
| --- | --- |
| 4K / 超宽桌面 | 对白、抽屉和字号设像素上限；立绘保持合理人物尺度，避免 UI 被拉到两端。 |
| 2K 桌面 | 使用窄幅对白和大工作台；默认同时展示多名在场角色。 |
| 1180px 以下 | 减少角色横向位移，隐藏次要场景文案，保持对白可读。 |
| 900px 以下 | HUD 收敛为图标，工作台接近全宽；对白改成单列。 |
| 640px 以下 | 同时只显示当前焦点角色；工作台为 bottom sheet；不照搬桌面多立绘站位。 |
| 低高度横屏 | 缩短对白高度，隐藏装饰性 scene caption，确保输入与停止按钮可见。 |

## 11. 交互状态机

| 状态 | 主要表现 | 可执行动作 | 离开条件 |
| --- | --- | --- | --- |
| `loading` | 背景占位、骨架 HUD | 返回 | 数据就绪或失败 |
| `role-required` | 强制角色选择模态 | 选择并绑定、返回 | `/role_bind` 成功 |
| `idle` | 当前对白或等待行动 | 继续、输入、打开工作台、纯净舞台 | 用户动作 |
| `paging` | 显示某段的第 N 页 | 补全、下一页、AUTO | 当前段读完 |
| `choice` | 展示快捷或建议行动 | 选择、自由行动 | 发送动作 |
| `composer` | 自由行动输入 | 发送、取消 | 发送或关闭 |
| `streaming` | thinking/tool/text 流式更新 | 停止、查看非阻断详情 | completed / error / cancelled |
| `stopping` | 停止按钮禁用并反馈 | 等待 | stop 结果返回 |
| `drawer` | 状态或日志大抽屉 | 浏览、关闭 | Esc、遮罩或关闭按钮 |
| `cinematic` | 仅背景与立绘 | 恢复 | H、Esc 或恢复按钮 |
| `error` | 保留已提交内容并展示错误 | 重试、编辑、返回 | 用户恢复 |

互斥规则：

- `drawer`、`composer`、`cinematic` 三者不同时存在。
- `streaming` 时可以查看日志或状态，但不能发起第二个普通 turn。
- 切换模型不取消当前生成，从下一 turn 生效。
- 纯净舞台不能绕开 `role-required`。

## 12. 键盘、焦点和 ARIA

### 12.1 快捷键

- `Enter`：对白补全 / 下一页 / 下一段；输入框聚焦时遵循输入规则。
- `1`–`3`：选择当前可见的预设行动。
- `S`：角色与世界状态。
- `L`：全部推演日志。
- `H`：进入或退出纯净舞台。
- `Esc`：按“菜单 → 抽屉 → 输入 → 纯净舞台”的当前层级关闭。

快捷键不得在 textarea、input、select 或 contenteditable 中抢占输入。

### 12.2 焦点

- 抽屉打开后焦点进入关闭按钮或标题后的首个可操作项。
- 抽屉内焦点保持在模态范围，关闭后回到触发按钮。
- 纯净舞台隐藏层全部 inert，恢复按钮成为唯一焦点入口。
- `:focus-visible` 必须清晰且不依赖颜色变化。
- 动画遵循 `prefers-reduced-motion`。

## 13. 边界与降级场景

| 场景 | 期望行为 |
| --- | --- |
| 没有背景 | 使用主题渐变或纯色，不阻断聊天。 |
| 背景媒体服务失败 | 保留最近成功背景或降级底色，显示非阻断提示。 |
| 没有立绘 | 使用头像、剪影或首字母占位。 |
| 多于可见站位的角色 | 聚焦在场且正在说话的角色，其余进入角色列或重叠安全布局。 |
| 说话人无法匹配角色 | 显示通用“角色”铭牌，不猜测角色 ID。 |
| assistant 标签不完整 | 解析器按现有 raw fallback 展示完整文本。 |
| 超长段落 | 强制换行并分页，绝不横向滚动或截断正文。 |
| 空历史 | 展示开场等待态；不生成虚构日志。 |
| 历史分页失败 | 保留当前窗口并允许重试。 |
| thinking 未持久化 | 刷新后不显示，日志说明仅包含可恢复记录。 |
| tool 内容过长 | 默认摘要，显式展开完整可公开内容。 |
| stop 成功 | 只有收到 cancelled 结果才显示“已停止”。 |
| stop 失败或 turn 已完成 | 以服务端最终状态为准，不本地伪造取消。 |
| Context 达到门禁 | 禁用普通正文输入并说明原因；斜杠命令仍保留现有可用规则。 |
| 玩家角色无效 | 展示不可取消角色绑定模态，不进入普通 turn。 |
| 状态表为空 | 显示“暂无字段”，不补默认字段。 |
| 关系字段为文本 | 原样换行展示，不解析成数值或进度条。 |
| TTS 尚无 message ID | 不提供播放；只对已提交 assistant message 派生。 |
| 移动端键盘弹出 | 输入区保持可见，舞台可缩退，停止按钮不能被遮挡。 |

## 14. 明确禁止事项

- 不把沉浸式页面嵌入 `SessionRoom.tsx`，不在标准页面组件树中堆叠沉浸式布局分支、HUD、抽屉或对白页状态。
- 不复制标准页面的 Session 业务链路；公共逻辑应抽入两个页面共同消费的前端领域 runtime/provider/hooks/service。
- 不把共享“中间层”实现为 Next.js `middleware.ts`。
- 不因沉浸分页新增或修改 SSE event type、payload、顺序与完成语义。
- 不按 SSE chunk 边界分页，不向 SSE、API、消息 metadata、数据库或 localStorage 写入视觉页码和分页结果。
- 不新增章节、支线、固定剧本进度作为 Session 必要字段。
- 不使用 `workspace + story_id + session_id` 三元定位会话内接口。
- 不由 WebUI 直接写玩家角色绑定。
- 不将 scene 当成普通 normal 状态表。
- 不用本轮 provider usage 覆盖下一轮 `context-preview` 圆环。
- 不将 TTS、背景、立绘或 Gallery 引用写进消息正文、消息 metadata 或 localStorage。
- 不把关系表固定为单一数字模型。
- 不从表名、角色名或 assistant 自然语言中猜稳定 ID。
- 不持久化完整内部 CoT，也不将“思考摘要”描述成完整推理。
- 不因媒体或工作台故障阻断普通聊天。

## 15. 设计验收清单

- [ ] 沉浸式体验拥有独立路由与页面组件树；标准 `SessionRoom` 不承载沉浸式 UI 分支和页面状态。
- [ ] 两个页面通过共享前端领域运行时复用 history、stream/stop、角色绑定、Composer、模型、Context Preview、status/media/TTS 等业务逻辑。
- [ ] 正文 SSE 的 event type、payload、顺序和完成语义保持现状，分页不依赖 chunk 边界。
- [ ] 同一份完整 SSE 正文在不同视口可以得到不同页数，但视觉页重组结果与原正文逐字符一致。
- [ ] 视口、字体缩放或方向变化会触发重分页；流式增长时已读页保持稳定，只重算未读尾部。
- [ ] 默认舞台没有章节或支线文案。
- [ ] 2K 与 4K 下关键文字清晰，对白没有无限拉长。
- [ ] 双角色时对白不遮挡面部和大部分上半身。
- [ ] 长对白可出现两次及以上“继续阅读”，且无字符丢失。
- [ ] 字体放大到 200% 后仍可分页、滚动和停止生成。
- [ ] 关系卡可同时显示多个数字字段、枚举字段和长文本字段。
- [ ] 关系字段更新日志逐字段可读，不再使用单一“好感 +N”。
- [ ] 多角色各自的状态表、关系、线索和待办能够在工作台中浏览。
- [ ] 日志只保留一个“全部”范围，并按 turn 分页。
- [ ] 纯净舞台隐藏层不可 Tab，且始终有恢复路径。
- [ ] 角色绑定、Context 门禁、stop、TTS 和模型切换沿用现有业务链路。
- [ ] 背景、立绘或媒体服务失败不影响发送与历史加载。
