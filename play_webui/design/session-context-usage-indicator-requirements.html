Play WebUI / SessionRoom requirement draft

Context 用量状态位

在 SessionRoom 输入框底部增加一个可点击的圆圈进度条，帮助玩家在发送前理解当前 context 占用。状态位优先展示 LLM provider 返回的准确 token usage；无法获取准确数据时使用后端 context-preview 或兜底估算，并清楚标注数据来源。

静态 HTML 不改运行逻辑 SessionRoom composer

需求摘要

  • 1状态位固定出现在 composer bottom 区域，靠近快捷键提示；底栏空间不足时换行显示，不挤压叙事风格控件。
  • 2圆圈主视觉表达“当前 prompt/context token 占模型 context 上限的百分比”，中心显示百分比；未知或异常时显示问号。
  • 3点击圆圈后在圆圈上方弹出详情，说明使用量、上限、数据来源、更新时间和异常/兜底原因。
  • 4context 上限必须从 llm_service 的 typed config 暴露给 Play API / WebUI，不允许前端写死模型窗口。
  • 5如果 provider 返回准确 usage，按准确值显示；否则用 context-preview 估算；再失败时使用兜底 token 估算并标明不精确。

数据优先级

1. Provider accurate usage LLM provider 返回 prompt_tokens / total_tokens 时优先使用。流式请求完成后可刷新为准确状态。
2. Context-preview estimate 未拿到 provider usage 时，使用现有 context-preview.v1 totals.tokenCount 作为后端估算来源。
3. Fallback estimate context-preview 失败、token counter 异常或服务不可用时，使用兜底字符估算，并在 UI 上标记为“估算/异常”。

实现约束

主实现不应让 Play WebUI 直接解析 YAML，也不能硬编码模型窗口。context 上限应由 llm_service 配置封装读取后，经 Agent service / Play API 暴露给 SessionRoom。

建议后续后端契约包含这些字段：usedTokens、contextLimit、ratio、source、accuracy、updatedAt、errorReason。接口可以新增在会话内链路下，也可以扩展现有 context-preview.v1 payload。

显示口径以“发送给 LLM 的 prompt/context token 占模型 context 上限”为主。completion token、cached token 和 provider 原始 usage 可放在详情里，不进入圆圈主百分比。
