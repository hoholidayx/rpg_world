玩家扮演角色选择

当前会话页会从故事角色中 fallback 出一个“你”的头像与名称，但缺少明确的“本局你扮演谁”绑定流程。本需求定义 session 级玩家角色选择，让新建会话、进入旧会话和会话内设置形成闭环。

问题背景

现状是“显示层猜测”，目标是“session 明确绑定”。

必须补齐
当前缺口

SessionRoom 使用故事角色 fallback 作为玩家身份，用户没有机会选择本局扮演角色。

用户感知

“你”的头像、名称和 IC 身份可能与真实想扮演的角色不一致，刷新后仍不可解释。

后续影响

后续 IC prompt、历史展示、角色视角记忆都需要稳定的玩家角色绑定。

目标体验

在开始游玩前确认“我是谁”，会话中保持身份稳定。

体验闭环
  • 1新建会话时，故事有已挂载角色则必须选择一个玩家扮演角色，确认后才进入 SessionRoom。
  • 2进入旧会话时，若已经绑定有效玩家角色则直接进入；若绑定缺失或数据异常，则强制进入角色选择页，不做静默兜底。
  • 3会话内设置菜单展示当前扮演角色，允许切换，但需要二次确认；切换只影响后续消息，不重写历史。
  • 4Agent 端在 send/stream 前也必须强校验玩家角色绑定，避免绕过 WebUI 入口后生成无身份消息。
  • 5其他渠道允许先创建空 session，但普通消息不会进入 LLM；Agent 固定返回编号角色列表，用户按格式选择后再开始剧情。
  • 6assistant 仍统一显示“旁白”，不会因为玩家选择了某个角色就拆分或冒充角色发言。

CLI / Telegram / Agent API 强校验下的补选协议

非 Web 渠道可以建立空 session，但空 session 只能处于“等待选择扮演角色”状态。任何普通输入在绑定前都不进入 LLM；Agent 固定返回当前故事已挂载角色的编号文本，用户回复 /role_bind 序号 后绑定或切换玩家角色，再发送 story first_message。

1

允许空 session

CLI、Telegram、Agent API create/ensure 都可以不传 playerCharacterId；session 创建成功，但状态是 player role required。

2

固定返回角色列表

首次普通输入或 /start_play 时返回所有 story mounted characters，带 1-based 序号、名称、简短摘要；该回复不经过 LLM。

3

用户选择格式

只接受 /role_bind 1 这种明确格式。无效序号、非数字、角色已取消挂载时，重复返回角色列表和错误提示。

4

绑定并开场

绑定成功后写入 playerCharacterId 与 snapshot，然后发送 story first_message；之后普通消息才允许进入 LLM。

数据结构与 API 后续计划

正式实现时建议作为 session 级绑定，而不是 UI 本地状态。

后续实现 
SessionSummary {

POST /sessions
body: { workspaceId, storyId, title?, playerCharacterId }
allow: playerCharacterId 缺省时创建空 session，状态为 player_role_required

Agent ensure/create session
body: { workspace_id, story_id, session_id?, title?, player_character_id? }
allow: player_character_id 缺省；返回 player_role_required

Telegram pending selection
state: chat_id -> { session_id, candidateCharacterIds[], awaitingRoleSelection: true }

CLI startup
config: workspace_id + story_id + optional session_id + optional player_character_id

PATCH /sessions/{session_id}/player-character
body: { playerCharacterId }

Agent send/stream guard
require: session.playerCharacterId 指向当前 story 已挂载角色
missing: 固定返回编号角色列表文本，不调用 LLM
bind command: /role_bind {index}
shared behavior: WebUI 设置面板选择角色与 /role_bind 走同一条绑定/切换业务能力
after bind: 写入 playerCharacterId + snapshot，并发送 story first_message
invalid binding: 409 player character invalid

验收标准

可验证
  • A本页能说明为什么需要玩家角色选择。
  • B角色卡选择为单选，未选择时确认按钮禁用。
  • C空状态说明需要先创建并挂载角色。
  • D绑定缺失或异常时，文档要求强制进入角色选择页，而不是 fallback 第一个角色。
  • E文档明确其他渠道可创建空 session，但绑定前只返回固定编号角色列表，不调用 LLM。
  • F文档明确 /role_bind 序号 是绑定或切换玩家角色的渠道文本协议，绑定成功后才发送 first_message。
  • G文档明确本页不实现后端能力。

落地阶段

建议
1

数据绑定

session 表增加玩家角色绑定，API 返回与修改该字段。

2

WebUI 接入

SessionCenter 和 SessionRoom 使用真实绑定角色，不再 fallback 猜测。

3

渠道选角协议

为空 session 增加固定角色列表回复与 /role_bind 序号 绑定逻辑；该步骤不调用 LLM。

4

Agent 强校验

send/stream 前校验 session 已绑定有效玩家角色；缺失返回选角列表，异常返回业务错误。

5

Agent 语义

通过强校验后，再将角色身份用于 IC prompt；不在本静态需求页实现。

确认切换扮演角色

切换后只影响后续 IC user 消息的显示身份和未来 prompt 语义。已有历史消息不会重写。
